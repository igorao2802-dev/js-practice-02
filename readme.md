## Ответы на контрольные вопросы. 
### **Вопрос 1:**   Почему мы используем textcontent вместо innerHTML для вывода простого текста? 
### **Ответ:** 
- <b>безопасность:</b> innerHTML может выполнить HTML‑код или скрипты, если в строке окажется вредоносный ввод.
textContent выводит только текст, не интерпретируя его как HTML;
- <b>производительность:</b> textContent работает быстрее, потому что не парсит HTML;
- <b>предсказуемость:</b> textContent всегда вставляет именно текст, а innerHTML может изменить структуру DOM.

## 

### **Вопрос 2:**  В чем разница между let и const? Когда какую переменную использовать?
### **Ответ:** 
- переменная const не может быть переопределена. Используется для значений, которые должны оставаться неизменными. Важно: содержимое объектов и массивов менять можно, но нельзя изменить саму ссылку;
- переменная let может быть изменена. Используется, когда значение должно меняться в процессе работы программы;
- таким образом, const применяется по умолчанию, если значение не должно меняться, а let — когда переменная действительно должна изменяться.

## Ответы на контрольные вопросы. 
### **Вопрос 3:**  Почему addEventListener считается более правильным подходом, чем inline-события в HTML?
### **Ответ:** 
- <b>разделение кода:</b> Inline-события смешивают HTML и JavaScript, что ухудшает читаемость. addEventListener позволяет держать логику в JS-файле;
- <b>несколько обработчиков:</b> через onclick можно назначить только один обработчик. addEventListener позволяет добавлять несколько;
- <b>гибкость:</b> поддерживает дополнительные параметры (once, capture, passive), что даёт больше контроля;
- <b>соответствие современным стандартам:</b> Inline-обработчики считаются устаревшими, а addEventListener — стандартный и рекомендуемый способ работы с событиями.

## Вопросы для самопроверки (Junior-интервью): 
### **Вопрос 1:**  Почему в современном коде мы избегаем использования var? 
### **Ответ:** 
Использование var считается устаревшим подходом, потому что он создаёт много проблем:
<br>1. var имеет функциональную, а не блочную область видимости, поэтому переменные «видны» за пределами блока {}, что приводит к неоднозначному их поведению.
<br>2. Переменные var поднимаются (hoisting) и инициализируются undefined, из-за чего ошибки проявляются только во время выполнения.
<br>3. var допускает повторное объявление, что повышает риск конфликтов имён и делает код менее надёжным.

##

### **Вопрос 2:**  В чем разница между addEventListener и написанием события прямо в HTML-теге (например, onclick="...")? 
### **Ответ:** 
<b>1. Разделение структуры и логики:</b> onclick смешивает HTML и JavaScript в одном файле, что делает код менее чистым.
addEventListener позволяет держать JS отдельно, что соответствует современным практикам.
<br><b>2. Можно назначать несколько обработчиков:</b> Через onclick можно повесить только один обработчик — новый перезапишет старый.
addEventListener позволяет добавлять сколько угодно обработчиков на одно событие.
<br><b>3. Гибкость и дополнительные возможности:</b> addEventListener поддерживает параметры (например, capture, once, passive), что даёт больше контроля над поведением события.
<br><b>4. Совместимость с современными стандартами:</b> addEventListener — часть стандартного DOM‑API и используется во всех современных проектах.
Inline‑обработчики (onclick) считаются устаревшим и менее безопасным подходом.

##

### **Вопрос 3:** Что произойдет, если попытаться изменить значение переменной, объявленной через const?
 
### **Ответ:** 
Переменная const создаёт константу, поэтому попытка присвоить ей новое значение приведёт к ошибке, скорее всего, "TypeError". JavaScript остановит выполнение скрипта и сообщит, что изменение значения константы невозможно. Изменять можно только содержимое объектов. Если const хранит объект или массив, менять саму ссылку нельзя, но менять внутренние свойства — можно.
